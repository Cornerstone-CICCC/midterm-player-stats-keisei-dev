# Reflection

I built a full-stack app called Player Performance Explorer. I used Astro for the front end, Express with TypeScript for the API, and PostgreSQL with the `pg` Pool. Users can look at player match stats, move between pages, and choose how many rows to show (10, 25, 50, or 100). They can also search by player name, filter by position or team, and sort the list. If you click a row, you can see the full details, edit the stats, or delete the record. There is also a page to create a new record. The Rankings page shows totals for each player. It uses a JOIN between `players` and `performances`, and it uses `COUNT`, `SUM`, and `AVG`. All SQL queries use parameters (`$1`, `$2`, ...) so the app is safer.

The original CSV file was one big flat table. Each row had player info, match info, and performance stats mixed together. The same player name and match date were copied again and again. That wastes space, and if you change one value (like a club name), you may need to change many rows. The script `setup_db.sql` splits the data into three tables. The `players` table has one row for each player. Its primary key is `player_id`. It stores things like `player_name`, `age`, `nationality`, `team`, and `position`. The `matches` table has one row for each match. Its primary key is `match_id`. It stores things like `match_date`, `stadium`, `city`, and `tournament_stage`. The `performances` table has one row for one player in one match. Its primary key is `id`. It has foreign keys: `player_id` points to `players`, and `match_id` points to `matches`. It also has a unique rule so the same player cannot have two rows for the same match. The stats (goals, assists, minutes, rating, and more) are in this table. Opponent and result are also in `performances`, because they belong to that player’s view of that match.

I had a few hard parts. First, I had to keep search, filter, and sort safe. Sort columns come from a whitelist, not from free text. Second, the `/filters` route must be above `/:id`, or Express thinks the word “filters” is an id. Third, an old API process on port 3000 caused errors until I restarted the server. If I improve this app later, I want dropdowns for player and match when creating a record, better checks for number values, and some automated tests for the API.

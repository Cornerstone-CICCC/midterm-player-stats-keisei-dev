# Reflection

I built a full-stack Player Performance Explorer for the FIFA World Cup 2026 dataset using Astro (SSR front end), Express with TypeScript (REST API), and PostgreSQL through the `pg` Pool. The app lets a user browse match performances with server-side pagination and a rows-per-page selector (10 / 25 / 50 / 100), search by player name, filter by position and team, and sort on meaningful columns. Each row opens a detail/edit page where stats can be updated or deleted, and there is also a create form for new performance records. A Rankings page shows per-player totals using a JOIN across `players` and `performances` with `COUNT`, `SUM`, and `AVG` aggregates. All database access uses parameterized queries.

The original CSV was flat: each row mixed player attributes, match attributes, and one player's stats for that match. That structure repeats the same player and match information many times, which wastes space and makes updates risky (for example, changing a player's club would require editing every related row). The provided `setup_db.sql` normalizes the data into three tables. `players` holds one row per player (`player_id` as primary key) with attributes that belong to the person/team context, such as `player_name`, `age`, `nationality`, `team`, `position`, and tournament totals. `matches` holds one row per match (`match_id` as primary key) with shared match context like `match_date`, `stadium`, `city`, and `tournament_stage`. `performances` is the fact table: each row is one player in one match, with a serial `id` primary key, foreign keys `player_id → players(player_id)` and `match_id → matches(match_id)`, a unique constraint on `(player_id, match_id)`, and the actual stats (goals, assists, minutes, rating, and many other metrics). Opponent and result fields live on `performances` because they are from that player's perspective in that match, not a global property of the match alone.

The main challenges were keeping server-side filtering/sorting safe (whitelist sort columns instead of interpolating user input), making sure `/filters` was registered before `/:id` so Express did not treat "filters" as an id, and coordinating the front end with a fresh API process after older servers were left running on port 3000. If I improved the project further, I would add dropdowns that load existing players and matches for create (instead of typing IDs by hand), stronger validation for numeric ranges, and automated tests for the API endpoints.
